Paper Review - “Disconnected Operation in the Coda File System”

1992

问题陈述、背景与研究动机

在分布式文件系统(distributed file system)中，处于断开连接状态的客户端会面临着文件资源不可用的窘境，在意外断开的情况下尤甚。

在此研究之前，已有较受普及的分布式文件系统NFS(Sun network file system)和AFS(Andrew File System)，但它们连接失败的时候会给用户带来较大的不便。

文章将此状态下的工作操作称为”disconnect operation”。

对于离线状态研究的相关工作：

Cedar file system 提供了不完全(rudimentary)的对disconnect operation的支持，但是没有提供hoarding、reintegration和conflict detection等操作。

FACE file system使用了称作”stashing”的方法来提供可用性，但是没有cache的同步。

PCMAIL system较完全地实现了disconnet operation，但是其过于依赖邮件语义(不能推广到文件系统)，而且可移植性较差。

此研究目标即为提高用户端在离线状态下文件系统的可用性。

####

coda文件系统设计：

overview:

假设：

  • 使用环境：少量的可信任的服务器和大量的不可信任的客户端
  • 客户端机器拥有磁盘等存储设备，并通过网络与服务器连接
  • 在某些时候，客户段会出现与服务器断开连接的情况

提供两种机制来保持可用性：

  • 在线状态下与服务器保持同步
  • 离线状态下通过合适的设计使得客户端能在一定程度上模拟服务器

设计框架讨论 & 因此产生的问题：

  • callback-based coherence
  • whole-file caching(简化实现)
  • placing of functionality on clients rather than severs.
  • 一级复制与二级复制

实现细节：

三种状态：hoarding, emulation, reintegration

hoarding - 囤积(缓存)文件:

hoarding的策略与可用性息息相关

难点：

  • 客户对文件的历史行为(reference behavior)不能(connot)具体预测其未来行为
  • 连接状态的不可预测(可能意外掉线或重连)
  • 断线状态下的cache miss代价难以估计
  • 需要考虑其他客户的行为
  • 有限大小的cache

coda的方案：

  • 按优先级划分(Prioritized Cache Management)

    划分依据：

    1. implicit：用户的历史行为
    2. explicit：HDB(per-workstation hoard database)

    hoard profiles

  • Hoard Walking

    定义平衡状态(equilibrium)：不存在对象A和B，A的优先级比B高，A在cache中而B不在cache中。

    平衡状态往往会被打破：因此设计Hoard Walking操作，周期性地(文章中周期为10min)维持平衡状态

    1. reevaluate the name-bindings of HDB entries
    2. reevaluate the priorities
    3. fetch and revict

    problem:

    Venus是基于callback break caching的，会出现文件的过期问题。

    妥协方案：一般文件在callback break后立马cache，对于目录文件则不cache。(不优美)

Emulation - 模拟服务器

当客户机掉线时，就需要进入emulation状态了。

Logging

coda内部实现了一种logging机制，称为replay log，每个replay log entry会记录对应的文件系统调用(e.g. write, open, close)和相应参数(和对象的版本状态)。

coda采用了一些优化方法来压缩replay log的大小，这对于disconnect状态下的机器是很重要的。

  • 优化1：注意到coda使用的是whole-file caching，对于 [open, write, write, … , write, close] 这样的操作流，我们不需要分开单独记录，只需要一个”store”即可。
  • 优化2：当一个新”store”加入时，原来的对于同一个文件的”store”记录可以舍去。
  • 优化3：考虑系统调用的逻辑关系，当某些特定的系统调用出现时可以舍去之前的被”overwrite”的记录，例如：unlink, truncate
  • 优化4：考虑互相抵消的操作，此时两个记录都可以舍去，例如rmdir和mkdir的抵消关系。

单机状态下的persistence：如何应对宕机重启？

时刻保持cache和相关数据结构保存在非易失性储存器里。

recoverable virtual memory (RVM)

Resource Exhaustion：如何应对存储空间不足的情况？

文章提供了几种未来会探究的方法(并未实现)：

  • 压缩cache -> 用时间换取空间
  • 为用户提供撤销的工具
  • 插个软盘 (“allow portions of the file cache and RVM to be written out to removable media such as floppy disks”)
Reintegration - 重整合

当客户机重新连上网络后，立马进入此状态，进行同步。

本机coda将replay log发送给AVSG (Accessible Volume Storage Group)，每个server对该replay log进行演绎，并执行相应的操作。

  • step1: 解析log，执行事务，给所有对应的文件加锁
  • step2: 检验replay log的合法性，例如：冲突检测，磁盘空间检测
  • step3: back-fetching
  • step4: 提交事务，解锁

如果replay成功：

  • 客户端释放replay log
  • 重新调整优先级

否则：

  • 保存现场(打个包)

细粒度的replay更能节省时间(在此实现中，replay是以volume为单位的)，作者打算利用precedence graph做进一步的优化。

Conflict Handling

write/write conflicts

每个文件的每个版本都会被标注一个唯一的storeid，replay之前会先check被replay的文件的storeid与客户机在disconnect前的storeid相同。

如果相同：直接replay并update一个新的storeid。

如果不同：

  • 对于store记录，直接abort
  • ……

检验&测试

  1. 同步耗时测试

  1. 缓存大小测试

3.冲突率测试

结论

disconnect operation 是一个很简单的想法 (“tantalizingly simple idea”)，但是实现起来有诸多细节需要考虑。

此工作可以看作是对write-back caching的一种扩展应用。

Intel Code Challenge Elimination Round F. Cyclic Cipher

思路：
对于每个数列的某个数字，其移到第一位的时间可以用$k \times len[i] + p[i]$来表示，即满足条件的时间满足$T = p(mod\ len)$
所以问题可以大概转成一坨同余方程，判断是不是有解的问题。
一开始根据tutorial的做法是开棵线段树维护方程的解，复杂度$O(NlogNlogN)$后面tle了，原因是快速乘法太慢，之后改成__int128，CF上总是编译错误。
然后看了些奇怪的代码发现其实根根本不需要维护方程的解，只需判断是否有解有可以了。
怎么判断有解呢？发现每个方程的$len \leq 40$，也就是最后合并出来的大模数分解质因数后都$\leq 40$，其实我们可以把所有的方程拆成$len$的所有以$\leq 40$的因子为模的方程，然后对于每类模相同的方程暴力判断即可。
学到了

容斥原理$\times 20$的版本

Codeforces Round #364 (Div. 1) D. Huffman Coding on Segment

思路
离线莫队后，这道题目的核心问题是：如何快速地模拟出huffman编码后的结果。
可以发现不同的出现频率最多只有$\sqrt{2n}$种，我们将出现频率$\leq \sqrt{n}$的和大于的分开讨论：
对于$val \leq \sqrt{2n}$的，如果这样的频率数有$X_{val}(X_{val} \leq 3)$个,那么我们在合并的过程中，一定会取出其中的$\frac{X_{val} - 1}{2} \times 2$个两两合并成$2\times val$.这样做一遍之后，小于$\sqrt{2n}$的数是$O(\sqrt{n})$级别的，而对于$>\sqrt{2n}$的数，因为在合并的过程中始终有$\sum val_i=n$所以这样的数也不会超过$\sqrt{n}$.之后暴力合并即可。
另外合并也需要利用到单调的技巧。

简要题解
离散化后暴力枚举小矩形。

G The Luckiest number
题目大意
给定一个数$n$,$n \leq 10^9$,问否找到一个数$k$,使得$nk$的每一位上都是$8$.如果有，输出$nk$的位数。
简要题解
符合条件的$nk$必然满足$(10^x - 1)/9 \ * \ 8 = nk$。通过一系列推导可知即求$10$关于$9n/gcd(n,8)$的阶，对欧拉函数枚举因子从小到大暴力判定。

H USTC campus network
题目大意
给定n个点，除开m条边被破坏意外，任意两点都有边相连，问与点1联通的点的个数。
简要题解
BFS.
利用bitset减少不必要的枚举！
